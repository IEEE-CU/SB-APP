## What Team 1 Owns (per agent.md Section 7.1)

All UI/UX ‚Äî React state, routing, form handling, Socket.io client, API calls. Non-negotiable constraints:

1. **Mock-first** ‚Äî build against mock APIs matching the contract; never block on Team 2
2. **Permission-gated UI** ‚Äî fetch `accessLevel` from `GET /api/v1/user/permissions`; never hardcode role strings
3. **Content sanitization** ‚Äî DOMPurify on all user-generated content (Community Hub, Reports, Announcements)
4. **Envelope compliance** ‚Äî every API call expects `{ success, data, meta }` from Section 2
5. **No direct service calls** ‚Äî never call `/api/v1/services/*`; always through Team 2 routes
6. **Commits on `T1` only**, merge to `develop`

## API Endpoints Team 1 Consumes

All prefixed with `/api/v1/`. These are defined in agent.md Section 2 (envelopes) and Section 7.2 (modules):

| Module | Endpoints | Frontend Pages |
|---|---|---|
| Auth | `POST /auth/login`, `POST /auth/register`, `POST /auth/change-password` | Login, Register, Change Password |
| Users | `GET /users`, `GET /users/:id`, `PATCH /users/:id` | Profile, User Management (admin) |
| Permissions | `GET /user/permissions`, `POST /access/check` | Permission gate throughout app |

| Announcements | `GET /announcements`, `POST/PATCH/DELETE /announcements` | Announcement feed, create/edit |
| Community | `GET/POST /community/messages` + Socket.io | Community Hub (real-time) |
| Health | `GET /health` | App startup status |

## Integration Adaptation Strategy

agent.md doesn't lock specific endpoint paths or model field names ‚Äî only the response envelope shape is frozen. The plan uses **mock-first, adapt-later**. Here's how integration works:

### Architecture for Easy Adaptation

1. **Centralized API layer** (`src/lib/api.ts`) ‚Äî all HTTP calls go through one file. When Team 2 ships real endpoints, update paths here only.
2. **Types as contract** (`src/types/models.ts`) ‚Äî expected shapes. When real APIs differ, update types here; TypeScript compiler catches every break across all pages.
3. **Service files per module** (e.g., `src/services/events.ts`) ‚Äî functions like `getEvents()`, `getEvent(id)`, `createEvent(data)`. Adapting one module doesn't touch others.
4. **Mock data mirrors types** (`mock/db.json`) ‚Äî uses same field names as frontend types. When real APIs differ, types and mock update together.

### Integration Checklist (when Team 2's APIs are ready)

For each module, when real endpoints become available:

- [ ] Confirm endpoint paths match (e.g., `GET /events` vs `GET /society/:id/events`)
- [ ] Confirm request body field names match
- [ ] Confirm response field names match (especially `id` vs `_id`, date formats)
- [ ] Confirm pagination query params (`page`, `limit`) and response meta shape
- [ ] Confirm error codes match agent.md Section 2.6
- [ ] Test with real auth token from Team 2's login endpoint
- [ ] Test permission gating with real RBAC from Team 5's `/user/permissions`
- [ ] Update mock to reflect real API once verified

### Known Unknowns (tracked until resolved)

| Item | Current Assumption | Where to Adapt |
|---|---|---|
| Model field names | `id`, `name`, `email`, `societyId`, `createdAt` etc. | `src/types/models.ts` + `mock/db.json` |
